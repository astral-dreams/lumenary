Codex review plan

Dialectical Engine, revised for build

Claude's Dialogues proposal is worth building. This review keeps the core idea intact, but tightens the artifact boundaries, evidence gates, scheduler fit, and publication path so the engine strengthens Lumenary without turning debate into performance.

Executive Judgment

Build it, but build it as a review and synthesis engine first, not as a direct mutation engine. The dialogue should create durable evidence about a claim's pressure points. Any score changes, status changes, and new synthesis ideas should pass through explicit ledgers and the existing promotion and originality gates.

The best part of Claude's plan is the shift from parallel output to structured cross-agent pressure. That is exactly the missing layer after the Frontier Engine: frontier steering finds the question, originality audit tests novelty, and dialogue tests whether two agents can force a better claim into existence.

The main correction is practical: the current codebase does not have a generic structured thinker interface yet. engine/thinker.py currently returns IdeaRecord. The dialectical runner needs a lower-level generate_structured() path that can use arbitrary schemas for challenge turns, crux extraction, and synthesis verdicts.

What Claude Got Right

The product idea is strong

Public dialogues make Lumenary's method legible. Readers can see how a claim is challenged, revised, or rejected instead of only seeing a final card.

The artifact stack is mostly right

A JSONL ledger, per-dialogue JSON, human-readable Markdown, convergence notes, and website pages are the right durable surfaces.

The outcome taxonomy is useful

Convergence, revision, demolition, standoff, and transcendence give the engine honest options besides forced synthesis.

The prompts have the right ethic

The prompts aim for real philosophical engagement, not cheap contradiction or politeness.

Main Critique

Issue Why It Matters Codex Revision
Challenge can become role-play Agents can manufacture disagreement even when the real issue is source weakness or scope confusion. Add a required steelman before objection. The challenger must restate the proponent's claim in a form the proponent would accept.
Direct score mutation is too risky The original idea record is historical evidence. Rewriting it after a dialogue would blur what the agent actually claimed. Write recommended score and status changes to reviews/dialogues/outcomes.jsonl. Promotion views can read them, but source records remain immutable.
Scheduler section is outdated The live LaunchAgent runs engine.parallel_research, which already runs Codex and Claude in parallel each 30-minute slot. Add dialogues as a sidecar after every N successful parallel runs, not as a replacement for alternating Codex and Claude slots.
The thinker interface is too narrow thinker.generate() returns an IdeaRecord, but dialogue turns need different schemas. Introduce engine/structured_thinker.py or extend thinker classes with generate_json(schema_path, prompt).
Transcendence is over-claimed A new phrasing from two agents is not enough to prove original thought. Rename the internal status to candidate_transcendence until the synthesis survives originality audit and later challenge.
Website may publish too much raw process Raw exchanges can include draft claims, unverified analogies, and brittle source use. Public pages should lead with a distilled dialogue brief and only expose full transcript sections when the underlying claims meet the current public thresholds.
Path dependency remains Frontier-driven pair selection can deepen only the clusters already favored by the Frontier Engine. Add a configurable wildcard quota: one orphan, low-priority, or cross-frontier tension after every few frontier-led dialogues.

Data flow

  1. The Frontier Engine produces or refreshes state/frontiers.json.
  2. The Dialogue Engine detects tensions inside the top frontier, plus any due wildcard tensions.
  3. The challenger performs a steelman, then a challenge.
  4. The proponent rebuts by conceding, defending, revising, and naming the crux.
  5. The challenger tests the revision and either accepts, narrows, or rejects the crux.
  6. A synthesis pass extracts the outcome, crux, evidence gaps, and optional candidate synthesis idea.
  7. The engine writes review artifacts and only appends a new synthesis idea if it has a complete source basis and a post-dialogue audit queue entry.
  8. Publication reads dialogue views, promotion decisions, and originality audits before rendering public claims.

Protocol Revision

The dialogue should have four reasoning moves, even if the public page still displays three main sections.

Turn Agent Required Output Purpose
0. Steelman Challenger steelman, fairness_check, claim_units Proves the challenger understood the claim before objecting.
1. Challenge Challenger strongest_objection, hidden_assumption, counter_model, new_primary_or_practitioner_source Finds the load-bearing weakness and introduces fresh evidence.
2. Rebuttal Proponent concessions, defense, revised_claim, crux, test_that_would_settle_it Separates real concessions from the claim that survives.
3. Counter-rebuttal Challenger revision_verdict, crux_verdict, remaining_risk Checks whether the revision actually improved the idea.
4. Synthesis Alternating synthesizer outcome, convergence_claim, unresolved_crux, candidate_synthesis, recommended_adjustments Creates the durable review artifact without overwriting history.

Tension Selection

Claude's signals are good, but I would add evidence quality and frontier pressure so the engine debates ideas that matter right now. 

{

  "concept_graph_contradiction": 0.18,
  "translation_strain": 0.12,
  "audit_anomaly_overlap": 0.12,
  "near_neighbor_conflict": 0.08,
  "source_overlap_with_divergent_conclusion": 0.06,
  "recency": 0.04
}

The detector should reject pairs when either idea lacks enough source basis, lacks an originality audit, has a blocked epistemic label, or was already demolished by a prior dialogue. It should also alternate direction so Codex and Claude both get chances to be proponent and challenger.

Artifact Rules

The dialogue record is a review object. It can recommend a revision, but it does not silently revise the original idea. Historical fidelity is more valuable than neatness.

  • reviews/dialogues/dialogues.jsonl is the append-only source of truth.
  • reviews/dialogues/outcomes.jsonl records recommended score and status changes.
  • hypotheses/ideas.jsonl receives a new synthesis record only for complete candidate syntheses, with idea_type: "synthesis" and parent dialogue metadata.
  • reviews/originality/audits.jsonl must receive a post-dialogue audit for any new synthesis before public promotion.
  • findings/convergences/ should be written only for convergence, revision, and candidate transcendence outcomes, not for standoffs or demolitions.
  • state/frontiers.json should consume dialogue outcomes during refresh: survived claims rise, demolished claims fall, unresolved cruxes become next tests.

Scheduler Fit

Claude's spec describes a scheduler that alternates Codex and Claude. The live system now runs both agents through engine.parallel_research in each 30-minute job, then distills, audits, refreshes frontiers, builds, commits, and deploys.

The safer production plan is:

  1. Add python3 -m engine.dialectic --detect-only for cheap agenda validation.
  2. Add python3 -m engine.dialectic --max-pairs 1 for manual runs.
  3. Add a wrapper counter in the LaunchAgent script or engine.parallel_research: after every fourth successful parallel run, call the dialectic sidecar.
  4. Do not run dialogues if a research lock is active, if the previous build failed, or if there is no complete originality audit for the candidate pair.
  5. Deploy dialogue pages only after npm run build, public-copy scan, and promotion checks pass.

Website Revision

Keep the public /dialogues/ page, but make it reader-facing and epistemically careful.

  • The index should show the question, agents, outcome, frontier, and one plain-language sentence about what changed.
  • The detail page should read like a journal symposium, not a chat log.
  • Each page should show clear labels: draft, review candidate, public claim, or synthesis ready.
  • Raw turn text should be available only when it does not expose unpublished or rejected claims as settled conclusions.
  • Finding pages should show "Debated" only when a dialogue is complete, with outcome and crux shown near the originality audit.
  • The Method page should eventually explain Dialogues as the layer after Frontiers: Frontiers chooses where to press, Dialogues applies cross-agent pressure.

Build Plan

Phase 1: Structured turn infrastructure

  1. Add a generic structured thinker helper that can call Codex CLI or Claude Code with an arbitrary JSON schema.
  2. Add dialogue schemas and tests for schema loading, JSON extraction, and sanitization.
  3. Add config/dialectic-rules.json for weights, cooldowns, wildcard cadence, minimum scores, and public rendering rules.

Phase 2: Tension detection

  1. Read hypotheses/ideas.jsonl, state/frontiers.json, graph/concept-graph.seed.json, and reviews/originality/audits.jsonl.
  2. Generate ranked TensionPair objects with explanatory signals, not just numeric priority.
  3. Write state/dialogue_agenda.json so the next run is inspectable before model calls are spent.

Phase 3: Dialogue runner

  1. Implement the steelman, challenge, rebuttal, counter-rebuttal, and synthesis calls.
  2. Persist every prompt, output, parsed JSON object, and failure in runs/dialectic-*/.
  3. Write append-only review artifacts under reviews/dialogues/.

Phase 4: Post-dialogue gates

  1. Queue or run an originality audit for any candidate synthesis idea.
  2. Refresh frontiers so unresolved cruxes become next tests.
  3. Record growth only with reader-facing bullets that match the current Growth style rules.

Phase 5: Website

  1. Add loadDialogues() and typed dialogue views to src/lib/content.ts.
  2. Create /dialogues/ and /dialogues/[slug].
  3. Add dialogue links to findings, frontiers, the map, homepage, and Method once the first real dialogue exists.

Phase 6: LaunchAgent integration

  1. Add the sidecar cadence after manual smoke tests succeed.
  2. Keep a --no-dialectic escape hatch for the half-hour job.
  3. Emit four macOS notifications: dialogue starting, challenge delivered, synthesis complete, artifacts written.

Acceptance Tests

  • python3 -m engine.dialectic --detect-only finds at least one explainable pair in the current corpus or exits cleanly with a reason.
  • A dry fixture run produces a complete DialogueRecord without calling live models.
  • A live run writes prompts, raw outputs, parsed JSON, Markdown, JSONL ledgers, and an outcome record.
  • No original idea record is overwritten by a dialogue outcome.
  • Any candidate synthesis receives an originality audit before it can be rendered as a public claim.
  • python3 -m unittest discover -s tests passes.
  • npm run build passes and public copy contains no banned punctuation.

Final Recommendation

Claude's plan is solid enough to build. The version I would build first is narrower and safer: an append-only Dialogue Review Engine with steelman gating, frontier-aware tension selection, generic structured model calls, post-dialogue originality audits, and public pages that show distilled dialectical movement rather than raw debate as settled truth.

Once that is working, we can let the half-hour LaunchAgent run it as a sidecar every few successful research cycles.
