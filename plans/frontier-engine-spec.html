Thesis

Lumenary should not merely produce many original findings. It should learn which questions deserve another pass, which claims are duplicates, which anomalies matter, and which public claims have actually advanced.

The Frontier Engine makes the loop genuinely recursive. It converts the critique layer into agenda control. Every research run begins from a live frontier brief instead of a broad standing prompt.

The Current Failure Mode

Weak feedback

Audits do not steer enough

Originality audits append strong follow-up instructions to state/next_directions.md, but the next Codex prompt still defaults to a broad cross-tradition finding request.

Duplicate pressure

Publication repeats winners

The publisher selects the best global public record. A strong older finding can keep winning, even when new work is happening elsewhere.

One-off risk

Findings accrete without enough lineage

The project can accumulate attractive insights without forcing each insight to face its closest prior art, anomaly class, source gap, and next test.

What The Frontier Engine Does

01

Ingest

Read ideas, audits, growth records, source cards, publication posts, and next directions.

02

Cluster

Group related findings into named frontiers by title terms, claim units, sources, audit status, and manual tags.

03

Diagnose

Identify what each frontier needs next: source grounding, anomaly test, merge, critique, practitioner test, or publication.

04

Prioritize

Score frontiers by promise, unresolved risk, recency, public value, and whether recent runs advanced them.

05

Brief

Write the next run prompt as a narrow assignment with source gaps, anomalies, and success criteria.

06

Govern

Prevent repeated publication unless the selected frontier has changed state or gained a new public contribution.

Non-Goals

Constraint

No hidden database as truth

Markdown and JSONL remain the durable source of truth. Any index is a rebuildable accelerator.

Constraint

No forced semantic AI clustering in v1

Start deterministic. Use transparent lexical, source, audit, and title overlap rules before adding embeddings.

Constraint

No automatic public claim escalation

The Frontier Engine recommends. Promotion still flows through the shared promotion gate and originality audit evidence.

Data Model

The engine writes two durable files: one machine-readable frontier ledger and one human-readable frontier brief. Both are derived from existing records and can be regenerated.

New Files

state/frontiers.json Canonical generated frontier state, sorted by priority and stable frontier ID.
state/frontiers.md Readable agenda map for humans, Codex, Claude, and future agents.
state/next_frontier_prompt.md The narrow prompt inserted into the next research run.
runs/frontier-events.jsonl Append-only log of selected frontier, score, action, and reason.

Status And Stage Values

Field Allowed Values Meaning
status active, cooldown, merged, retired, blocked Whether the frontier can receive the next run.
stage seed, probe, rubric, source_grounding, practitioner_test, synthesis, publication The kind of work the frontier most needs.
recommended_next_action advance, merge, falsify, source_ground, build_rubric, distill, publish, retire The next run assignment and publication behavior.

Algorithms

Clustering

Transparent v1 grouping

Use deterministic matching so the research agenda is explainable.

  1. Normalize title, claim, source basis, and audit claim units.
  2. Extract key phrases from titles, audit unlike statements, and near-neighbor sources.
  3. Give strong weight to shared manually meaningful phrases: translation strain, inferential gap, remainder, effort, attention.
  4. Merge ideas when they share a phrase cluster, at least one source family, or audit instruction overlap.
  5. Keep a manual override file optional later: state/frontier-overrides.json.
Prioritization

What gets the next run

Priority should reward promise and unresolved pressure, not just high self-scores.

priority =
  0.22 * strongest_generativity
+ 0.18 * audit_pressure
+ 0.16 * public_value
+ 0.14 * unresolved_anomaly
+ 0.12 * source_gap_importance
+ 0.10 * cross_agent_signal
+ 0.08 * recency
- 0.18 * duplicate_penalty
- 0.12 * cooldown_penalty

Audit Pressure

Signal Effect
audit_incomplete High pressure to complete audit, low publication permission.
known or renamed Recommend merge or retire unless a useful extension remains.
extended Ask for exact unlike statement, prior-art comparison, or a sharper test.
novel_synthesis Ask for rubric, anomaly test, practitioner test, or synthesis draft.
candidate_discovery Ask for adversarial challenge, human review, or public synthesis.

Duplicate Publication Guard

A frontier can publish again only when one of these is true:

  • A new idea in the frontier passes the public claim gate and was not previously published.
  • An audit changes the frontier stage, status, or confidence materially.
  • The next run resolves a named anomaly, produces a rubric, or creates a synthesis-ready public artifact.
  • The publication reason is explicitly journal, frontier-summary, or human-selected.

Prompt Shape

Each research run should receive a narrow frontier brief. It should feel like a serious research assignment, not a theme. 

# Active Frontier Brief

Frontier: Translation strain as a test of convergence
Recommended action: build_rubric
Why now: Recent audits found strong promise, but the claim needs a decision rubric.

Core claim:
Agreement matters most when it reveals what each side must bend.

Closest prior art:
- Comparative theology and anti-essentialist comparative religion already challenge easy convergence.
- The possible new contribution is a workflow that changes decisions, not the general warning.

Required task:
Build a one-page translation-strain rubric with fields for claim unit, role, ontology, agency, practice aim, authority, verification, what is bent, what is dropped, what is added, and whether role-similarity survives.

Required tests:
- Test the rubric on Atman versus anatta.
- Test it on apophatic darkness versus Buddhist emptiness.
- Test it on fana or baqa versus nondual realization.

Success criteria:
- Name one case where ordinary convergence scoring would say yes but the rubric says no.
- Name one case where high strain remains productive rather than disqualifying.
- Keep spiritual claims and scientific claims separate.
- Return exactly one structured idea record.

Public Page

The public route should show Lumenary as a live research mind: not only what it believes, but what it is testing, correcting, merging, and refusing.

Route

/frontiers/

Initial page can be serverless-static through Astro, reading state/frontiers.json at build time.

  • Active frontier cards sorted by priority.
  • Stage, status, latest audit status, and next test.
  • Linked ideas, audits, sources, findings, and publication posts.
  • Clear distinction between public claims and unresolved research questions.
Tone

Reader promise

The page should say: here is what I am learning, here is what could prove me wrong, and here is what I will test next.

  • No scoring language in card headlines.
  • No private source paths in reader-facing text.
  • Epistemic status visible where claims appear.
  • Plain language summaries for non-specialists.

Card Fields

Field Reader Text Source
Title What this question is really about. frontier.title
What I am testing One plain sentence from the core claim. frontier.core_claim
What could change my mind One anomaly or falsifier. frontier.open_anomalies, frontier.tests
Next move One concrete next research action. frontier.recommended_next_action
Evidence trail Links to findings, audits, source cards, and daily posts. idea_ids, audit_ids, publication_paths

Implementation Plan

Phase 1

Build the private Frontier Engine

Create engine/frontier.py, dataclasses, deterministic readers, grouping, scoring, Markdown rendering, JSON rendering, and CLI entry point.

  • Command: python3 -m engine.frontier refresh
  • Outputs: state/frontiers.json, state/frontiers.md, state/next_frontier_prompt.md
  • Tests: fixture-free tests using temporary JSONL and Markdown records.
Phase 2

Wire prompt steering

Refresh frontiers before generation. Insert the selected frontier brief into both Codex and Claude prompts. Preserve the user's explicit --focus as an additional constraint.

  • Add frontier_brief parameter to prompt builders.
  • Add --no-frontier escape hatch for manual broad runs.
  • Log selected frontier to runs/frontier-events.jsonl.
Phase 3

Wire publication governance

Teach the publisher to prefer newly advanced frontiers and avoid repeated daily posts from the same unchanged record.

  • Track publication history by idea ID and frontier ID.
  • Prefer frontier.last_advanced_at over raw global score.
  • Allow explicit human override later through state/publication-overrides.json.
Phase 4

Build internal review page

Add src/pages/frontiers/index.astro and content loaders. Start noindex if the copy still exposes too much internal machinery.

  • Cards for active, blocked, merged, and retired frontiers.
  • Filters by status, stage, agent, and originality status.
  • Links into findings, audits, daily posts, and sources.
Phase 5

Promote to public story

Polish the page for readers. Make the frontiers route part of the main navigation only when the language is plain, honest, and non-technical.

  • Add route to navigation.
  • Add structured metadata and RSS affordances later if useful.
  • Use it as the living research agenda for daily publication.

Acceptance Tests

Agenda shift

Given recent audits with next-loop instructions, the next prompt contains the top frontier's required task and tests.

Merge detection

Known duplicate findings such as repeated translation-strain records are clustered into one frontier with a merge recommendation.

Publication guard

Repeated daily publication of an unchanged older winner stops unless the frontier advances or a human override is present.

Audit carry-forward

An audit instruction becomes a visible next prompt, a frontier blocker, or a test item. It cannot vanish into a long Markdown tail.

Risk Register

Risk Impact Mitigation
Over-clustering distinct ideas The engine may flatten real disagreements. Preserve all idea IDs, agents, critiques, and audit disagreements. Add manual split overrides after v1.
Under-clustering duplicates The loop keeps rediscovering the same insight. Use title phrases, audit near-neighbor overlaps, and publication history penalties.
Prompt narrowing too much The system may become bureaucratic and lose creative range. Keep one exploration slot in the scoring formula and allow --no-frontier.
Public page exposes draft claims Readers may treat unresolved research as settled. Start internal or noindex. Use plain status labels and "what could change my mind" copy.
Scoring feels too mechanical The project may optimize for bookkeeping instead of insight. Keep the score explainable and make the final action text visible. The score chooses attention, not truth.

Open Questions For Review

Should /frontiers/ be public on day one?

Recommendation: build the route first, mark noindex, and add it to navigation after one day of clean frontier state.

Should frontier selection ever override explicit user focus?

Recommendation: no. Explicit user focus should constrain the frontier brief. The engine should only choose among compatible agendas.

Should embeddings be added immediately?

Recommendation: no. Start transparent. Add embeddings only if deterministic clustering misses enough cases to justify them.

Should the Frontier Engine rewrite idea records?

Recommendation: no. It should generate derived state and publication decisions. Original ideas and audits remain immutable records except for existing upsert behavior.
