AEO template audit

The Lumenary should answer before it explains.

This audit reviews the current local build of thelumenary.org after the latest template changes. The goal is a clean set of website templates where answer engines, search engines, and human readers can quickly understand what each page is, what claim it makes, what evidence supports it, and where to go next.

Date: May 26, 2026 Pages built: 70 Findings pages: 52 Audit target: templates first

Executive Read

7.2/ 10

Current AEO Readiness

The site has strong raw material: original findings, concise summaries, structured data, clean URLs, source cards, audits, and living maps. The main gap is template discipline.

8.4/ 10

Individual Findings

Finding pages are the strongest AEO surface because they have Article schema, breadcrumbs, FAQ blocks, At a Glance sections, originality audits, source basis, and critique.

5.8/ 10

Section Pages

Section pages vary too much. Insights, Journal, Growth, and Frontiers need their own schema, answer blocks, breadcrumbs, and consistent page purposes.

What AEO Means Here

For The Lumenary, AEO is not tricking AI systems into quoting the site. It is making each page easy to cite, summarize, and trust. Every important template should have a visible answer, a clear entity, a clean claim, structured support, and a route to deeper evidence.

  • Answer first: each page should answer its main question in one plain paragraph near the top.
  • Entity clarity: the page should state whether it is a project, collection, map, research method, insight card set, journal, or finding.
  • Evidence path: each claim should lead to source basis, audit notes, publication status, or related findings.
  • Schema mirrors visible content: structured data should describe what readers can actually see on the page.
  • Snippet discipline: titles and meta descriptions should be short, unique, and direct enough to survive search rewrites.

Sitewide Findings

Strength: strong base layout

The base template already emits canonical URLs, robots tags, RSS, Open Graph, Twitter metadata, WebSite schema, Organization schema, and GA instrumentation. This is a good foundation.

Gap: missing universal breadcrumbs

Individual findings have BreadcrumbList schema, but section pages do not. Add a breadcrumb schema helper to all section pages and render a small visible breadcrumb trail where it helps.

Gap: schema is hand-built per page

Schema should be generated from a small library of helpers: WebPage, CollectionPage, Article, BreadcrumbList, FAQPage, ItemList, Dataset, SoftwareApplication, and HowTo.

Risk: FAQ schema overuse

FAQPage is useful when questions are visible and natural, but Google limits FAQ rich result visibility. Keep FAQ blocks where they help readers; do not use them as decoration.

Gap: no Open Graph image strategy

Most pages have no page-specific social image. Add a default Lumenary image and generated images for Findings, Map, Journal, and Frontiers when possible.

Gap: stale route handling

The Agent page has been removed. Add a deliberate redirect or gone rule for /agent/ so crawlers do not find a confusing stale page.

Template Audit

Home Good entity, needs sharper answer hierarchy

The home page has WebPage and FAQPage schema, one H1, a strong source foundation, and a clear project description. It now avoids the weak recent-finding panel.

Add a stronger first answer: "The Lumenary is..." followed by "What it studies," "What it publishes," and "How to read claims." Move Promoted Public Claims lower or rename it in human language.

Insights Human page is good, AEO layer is thin

Insights is intentionally simple and should stay that way. It currently only receives WebSite and Organization schema from the base layout.

Add CollectionPage and ItemList schema for the visible insight cards. Add one concise visible answer block explaining that Insights are short distillations from Findings, not separate claims.

Journal Needs journal-specific schema

The Journal index is clean, but it lacks CollectionPage or Blog schema. Individual Journal posts should carry BlogPosting or Article schema with datePublished, dateModified, author, publisher, and breadcrumb.

Keep the voice simple. The AEO addition should be invisible to the tone: one short line saying these are daily reflections written by The Lumenary as "We."

Growth Excellent concept, under-described for machines

Growth is a distinctive page. It tracks growth in knowledge and growth in method, but the template does not yet describe that as a structured recurring record.

Add Dataset or CollectionPage schema, ItemList entries for day/week/month records, and a visible answer that says Growth records what The Lumenary learned and how its way of seeing changed.

Frontiers Strong AEO candidate, needs a definition block

Frontiers is valuable because it shows live research agendas, anomalies, tests, source gaps, and next moves. It currently has CollectionPage schema but no FAQ or breadcrumb schema.

Add a direct "What is a frontier?" answer at the top, ItemList schema for active frontiers, and breadcrumb schema. Keep the cards, but make each card answer one question: what is being tested next?

Map Good Dataset schema, needs SVG accessibility and entity links

Map has CollectionPage, Dataset, and FAQPage schema. It is one of the better AEO pages because the page has a clear purpose and named maps.

Add title and desc tags inside each SVG, expose each map as ImageObject or hasPart, and ensure every major bubble links to the relevant Finding, Source, Frontier, or concept.

Findings Index Good schema, noisy heading structure

Findings has CollectionPage, ItemList, FAQPage, and a strong direct answer section. It currently emits many H2 headings because each finding card uses an H2.

Keep one H1 and section-level H2s, then make finding card titles H3s. Add filters as crawlable links or explicit section anchors if topic discovery matters.

Individual Findings Best template, needs tighter metadata

Finding pages have Article, BreadcrumbList, FAQPage, At a Glance, originality audit, source basis, critique, scores, and next directions. This is the strongest page type.

Tighten meta descriptions to 140 to 165 characters, set real dateModified when audits or summaries change, add citations or isBasedOn to Article schema, and add relatedLink entries for connected findings.

Method High value, but too many jobs on one page

Method has HowTo, SoftwareApplication, and FAQPage schema. It is useful, but it also carries system, frontier, agent, publication, and runtime content.

Keep Method as the single home for the process now that Agent is removed. Make it cleaner: first answer, exact loop, frontier steering, promotion gates, storage, scheduler, and validation.

Sources Needs source catalog schema

Sources is a core trust page, but it is not prominent in the nav and should carry stronger data-catalog semantics.

Add Dataset or DataCatalog schema, source type filters, canonical source cards, and an answer block explaining how sources constrain claims.

Recommended Template Architecture

1. Create a schema helper layer

Add a small TypeScript module that exports helpers for base graph nodes, breadcrumbs, ItemLists, CollectionPages, Articles, FAQs, Datasets, BlogPostings, and HowTos.

2. Create reusable answer components

Add components for AnswerPanel, PageFaq, Breadcrumbs, PageMeta, RelatedLinks, and EvidenceTrail. The labels should be human, not technical.

3. Define per-template AEO contracts

Each page type should declare required title, description, H1, answer paragraph, schema types, breadcrumb path, related links, and evidence links.

4. Add build-time validation

Add an npm script that fails if a public page lacks a canonical URL, H1, meta description, schema graph, or valid internal links.

Fix Plan

  1. Base layout: add optional page image metadata, shared breadcrumbs, stable schema IDs, WebSite SearchAction only if a real site search exists, and consistent publisher identity.
  2. Section pages: add CollectionPage schema, ItemList schema, breadcrumb schema, and one reader-facing answer paragraph to Insights, Journal, Growth, Frontiers, Sources, and Daily.
  3. Finding pages: tune meta descriptions, set dateModified from the source file or audit, add citation/isBasedOn from source_basis, add relatedLink, and keep At a Glance as the primary answer.
  4. Heading cleanup: convert repeated card H2s to H3s on Findings and other card grids. Reserve H2 for page sections.
  5. Map page: add accessible SVG titles and descriptions, link bubbles, and expose each SVG map as a described part of the page.
  6. Removed Agent route: decide whether /agent/ should 301 to /research/ or return 410 gone. I recommend 301 to Method for now because the runtime material lives there.
  7. Validation: add a local audit script that checks titles, descriptions, H1s, schema, canonical URLs, breadcrumb coverage, no stale routes, and internal link health before deploy.

Implementation Order

Pass 1: Template primitives

Build schema helpers, answer blocks, breadcrumbs, and validation. This prevents future pages from drifting.

Pass 2: Finding detail

Improve the highest-value pages first: Finding pages should become citation-ready answer documents.

Pass 3: Section pages

Clean Insights, Growth, Journal, Frontiers, Map, Sources, Findings index, and Method against the same contract.

Pass 4: Content tone

Keep Insights and Journal simple. Add AEO structure around them, not academic language inside them.

Pass 5: Rich media

Add default OG images, finding-specific images when possible, and accessible SVG map metadata.

Pass 6: Monitoring

Wire the local analytics dashboard to watch indexed pages, AI referral traffic, top landing pages, source queries, and pages with low engagement.

Source Notes

This audit follows Google Search Central guidance that AI features use the same foundational search requirements as Google Search: pages must be indexable, eligible for snippets, technically accessible, and useful to people.

Structured data recommendations are constrained by Google guidance that schema should match visible content and should not mark up irrelevant or hidden material.
